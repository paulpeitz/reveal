Formatierung = Layout

Beispiel: Struktur + Inhalt

Idee: William Tunnicliffe (1967)
Tunnicliffe: Trennung des (informationellen) Inhalts des Dokuments von seinem äußeren Format

Strukturdefinition für eine Klasse von Dokumenten
Instanz enthält Struktur und Inhalt

Vorteile:
  • Programm prüft, ob die Dokumente der vorgegebenen Struktur entsprechen -> Validierung
  • logische Strukturierung unterstützt weitere Anwendungen der Dokumentenverarbeitung, z. B. Kapitelnummerierung berechnen, Inhaltsverzeichnis oder Übersicht verwendeter Begriffe automatisch generieren.

Beispiel: Formatierung (fiktiv) – style sheet

<buch>:DIN A4, linker Rand 2cm, rechter Rand 2,5cm
<titel>:Arial, 20pt, fett, linksbündig
<zusammenfassung>: drucke "Preface" in Arial, 14pt, fett, linksbündig;
Times New Roman, 11pt, normal, linksbündig
<teilüberschrift>: Arial, 16pt, normal, linksbündig
<überschrift>: Arial, 14pt, fett, linksbündig drucke <überschrift-nummer>
<abschnitt>: Times New Roman, 11pt, normal, Blocksatz …

Style sheet - Anweisungen oder Regeln, wie die Inhalte in einem strukturierten Dokument formatiert werden sollen

Vorteile:
  • Verschiedene Dokumente mit ein- und demselben Stylesheet einheitlich formatieren.
  • Ein- und dasselbe Dokument durch Kombination mit verschiedenen Stylesheets passend zum Anwendungszweck formatieren.
  • parallel arbeiten: einer die Texte, ein anderer die Stylesheets.

Begriffe

  • Tag: Textmarkierungen (markup= dt. Auszeichnung)
  • Element: hat öffnende und schließende Tags plus den Inhalt dazwischen
    • Elemente ohne Inhalt sind leere Elemente
    • HTML: es gibt auch Elemente ohne schließenden Tag und damit ohne Inhalt
  • Attribut: zusätzliche Eigenschaft des Elements
Markup eine Folge von Zeichen, die an bestimmten Stellen in einem Dokument eingefügt werden, um die Form der Darstellung, des Druckes oder der Struktur der Dokumente zu beschreiben

Standardisierung: SGML und DSSSL (ISO: 8879)

1978-1986: SGML (Standard Generalized Markup Language), Sprache zur Strukturdefinition von Dokumenten
sehr mächtig (ca. 400 Seiten Dokumentation)
zugehörige Formatierungssprache: DSSSL (Dynamic Style Sheet Specification Language)

DTD – Strukturdefinition
Sprache zur Beschreibung der Strukturdefinition eines Dokuments

Metasprache zur Beschreibung von Auszeichnungssprachen (Markup-Sprachen)

    Web-Seiten (deren Struktur) müssen beschrieben werden

    • 1995: Browserkrieg zwischen Netscape, Microsoft, SUN etc.
      • In schneller Folge wurden neue Versionen von HTML definiert

  • Eine Datei mit HTML-Text ist (bzw. war) eine einfache Textdatei
  • Erstellung mit einem üblichen Texteditor (z.B. mit Notepad, UltraEdit)
  • Ursprünglich hatte HTML nur einige wenige einfache Textmarkierungen (Tags).
  • Hypertextsprache, in der Text mit Querverweisen durchmischt wird

Die 3 Zeilen vor den eigentlichen Daten heißen Prolog
  • XML-Deklaration:
    • Version ist Pflicht. Die anderen in Reihenfolge. (Einsatz der Version 1.0 wird empfohlen, da die 1.1 nicht in allen Fällen rückwärtskompatibel ist.)

White Space - Leerraumzeichen

  • White Space = Leerzeichen, Tabs, Zeilenumbrüche

  • grundsätzliche Behandlung:
    • innerhalb von Elementen darf "White Space" entfernt werden
    • "White Space" im Inhalt bleibt unverändert

  • spezielles Attribut "xml:space"
    • xml:space="preserve": White Space innerhalb von Elementen bleibt
    • xml:space="default": ... kann entfernt werden
"innerhalb von Elementen": bei Elementen, die wiederum Elemente enthalten (z.B. bei Einrückungen) wird der die inneren Elmente umgebende Whitespace ignoriert

Document Type Definition – DTD

Eine DTD ist eine Auflistung der möglichen

  • Elementtypen
  • deren Beziehung zueinander (Verschachtelung)

  • deren Attribute

  • und von sogen. Entities (eine Art von Makro)

DTD: Regeln für Elementnamen

  • Name muss mit Buchstaben oder "_" Unterstrich beginnen
  • Nach dem ersten Zeichen sind zusätzlich Zahlen, "-" Bindestrich und "." Punkt erlaubt
  • Keine Leerzeichen
  • Groß- und Kleinschreibung wird unterschieden
  • Namen dürfen nicht mit einer Kombination der Buchstaben "xml" beginnen
  • Zwischen dem Elementnamen und der öffnenden Klammer "<" darf kein Leerzeichen stehen (vor der schließenden spitzen Klammer ">" darf ein Leerzeichen sein)
  • ":" Doppelpunkte sollten nur mit Namensräumen verwendet werden

DTD: Elementtyp-Deklaration

  • legt Namen und Inhalt von Elementen fest
    <!ELEMENT Name Inhalt>

  • Sequenz
    <!ELEMENT Rezept (Titel, Zutaten, Zubereitung)>

  • Alternativen
    <!ELEMENT Block (P | Table | IMG)>

DTD: Elementtyp-Deklaration

  • Kardinalitäten 0..1, 1..*, 0..*
    0..1: <!ELEMENT Name (Vorname, MidInitial?, Nachname)>
         (MidInitial kommt höchstens einmal vor)

    1..*: <!ELEMENT Liste (Überschrift, Item+)>
         (Item kommt mindestens einmal vor)

    0..*: <!ELEMENT StückListe (Artikel*)>
         (Artikel kommt beliebig oft vor (inkl. gar nicht))

         alle Inhalte ohne Angabe der Kardinalität kommen genau einmal vor

DTD: Elementtyp-Deklaration

  • Text als Inhalt (Parsed Character DATA)
    <!ELEMENT Vorname (#PCDATA)>

  • gemischter Inhalt
    <!ELEMENT P (#PCDATA | EM | (B, A)+)*>

  • beliebiger Inhalt
    <!ELEMENT Div ANY>

  • leere Elemente
    <!ELEMENT HR EMPTY>

DTD: Attributlisten-Deklaration

  • definiert
    • Zugehörigkeit zu einem Elementtyp
    • Attributnamen
    • Typ der Attributwerte
    • Default-Werte
  • IMPLIED -> Optional
  • REQUIRED -> Müssen

CDATA – ohne parsen, Parser ignoriert Sonderzeichen, sinnvoll wenn Text viele enthält, schneller zu übertragen

DTD: Attributlisten-Deklaration

Mögliche Attribut-Typen:

  • CDATA: Zeichenkette
  • ID: eindeutiger Name innerhalb eines Dokuments
  • IDREF: Referenz auf ein Element mit ID
  • IDREFS: Liste von IDREFs
  • ENTITY: Name eines vordefinierten externen Entities
  • ENTITIES: Liste von ENTITYs
  • NMTOKEN: ein Name (als Token)
  • NMTOKENS: Liste von Namen (Token)
  • Aufzählung
  • nur ein ID-Attribut je Element; dieses muss IMPLIED oder REQUIRED sein
  • IDREF: ermöglicht eine Referenz auf Inhalte, damit diese nicht wiederholt werden müssen, Attributwert muss gleich einer ID sein
  • ENTITY-Attribute dürfen ein an anderer Stelle in der DTD deklariertes Entity referenzieren, Entities erlauben den Einsatz von Textbausteinen -> ∧nbsp;
  • NMTOKEN behandelt Attribut-Werte nicht als String, sondern als eigenständiges Token (Menge der Token ist offen)
  • NMTOKEN erlaub auch Namen mit führenden Ziffern

DTD einbinden

XML-Dokument/Instanz verweist auf seine DTD
  • Extern (referenziert über URI)
    <!DOCTYPE Messwerte SYSTEM "werte.dtd">
    <!DOCTYPE html Root
      PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN"
        "xhtml1-strict.dtd">

  • Intern (innerhalb des XML-Dokuments)
    <!DOCTYPE List [
      <!ELEMENT List (Item)*>
      <!ELEMENT Item (#PCDATA)>
    ]>
das erste Wort nach DOCTYPE ist der Name des Root-Elements dieser Dokumentklasse

bei DOCTYPE "PUBLIC" gibt die erste URI eine "bekannte" DTD an, die das Werkzeug selbständig finden kann (ist dies doch nicht möglich, so ist eine zweite URI anzugeben, die direkt auf die DTD zeigt)

DTD: Nachteile

  • eine DTD ist selber kein XML-Dokument
    (gesonderte Syntax; Erbe von SGML)

  • Wiederverwendung bzw. Erweiterung ist schwierig
    • keine Möglichkeit nur einen Teil einer DTD zu verwenden
    • kein Vererbungsmechanismus
  • keine Datentypen im eigentlichen Sinne

XML: Wohlgeformt und Gültigkeit

Dokument ist ...

  • wohlgeformt (well-formed)
    • genügt der XML-Syntax
    • genau ein Wurzel-Element
    • korrekte Schachtelung
    • keine DTD notwendig

  • gültig (valid)
    • Dokument ist wohlgeformt
    • enthält DocumentType Declaration
    • es existiert DTD
    • Inhalt entspricht dieser DTD
